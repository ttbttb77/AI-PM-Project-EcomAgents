## 2. 产品愿景与目标

### 2.1 产品愿景

**打造电商领域的"AI 数字员工团队"，让每个电商商家都能拥有 7×24 小时自主运转的智能运营力量。**

电商多 Agent 系统基于 Multi-Agent 协同架构，将智能客服、内容创作、运营自动化、数据分析等核心电商能力封装为独立且可协作的 Agent，通过 MCP 协议标准实现与淘宝、京东、抖音等主流电商平台的统一对接，帮助商家实现如下转变：

- **从"人找工具"到"Agent 主动执行"**：商家无需学习复杂的工具操作，Agent 自主理解意图并完成任务。
- **从"单点提效"到"全链路自动化"**：覆盖客服、商品、营销、财务全流程，打破数据孤岛。
- **从"一次性交付"到"持续自进化"**：Agent 根据实际运行反馈自我优化，越用越懂业务。

### 2.2 产品目标

#### 2.2.1 商业目标

| 目标 | 衡量指标 | 里程碑 |
|------|---------|--------|
| 客服成本大幅降低 | 客服人力成本降低 40%+ | MVP 上线后 1 个月内达成 |
| 内容生产效率飞跃 | 商品内容生成效率提升 4 倍+ | MVP 上线后 2 周内达成 |
| 运营人效提升 | 运营人员日均操作时间降低 50% | Phase 2 上线后 1 个月内达成 |
| 平台覆盖广度 | 支持淘宝/京东/抖音三大平台 | Phase 3 达成 |

#### 2.2.2 模型目标（🆕 新增）

| 目标 | 衡量指标 | 目标值 |
|------|---------|--------|
| 意图识别准确率 | 标准评测集 | ≥ 90% |
| RAG 检索召回率 | Top-5 召回 | ≥ 95% |
| 回复置信度 | Agent 自评与实际一致性 | 偏差 < 10% |
| 内容生成采纳率 | 运营直接使用或微调后使用的比例 | ≥ 70% |
| 安全合规一票否决率 | 评测集中违反安全规范的比例 | < 0.1% |

### 2.3 核心价值主张

> **MVP 阶段（2 周内上线）**："客服成本降低 40%，内容生产效率提升 4 倍。"
>
> **完整系统（6 个月内）**："覆盖售前-售中-售后全链路，实现 60% 日常运营工作自动化。"

### 2.4 为什么用多 Agent + 大模型解决（🆕 新增）

> **这是 AI 产品 PRD 区别于传统 PRD 的核心问题——必须回答清楚：为什么这个场景必须用大模型 + 多 Agent 架构解决，而不是规则引擎、传统 ML 或纯人工？**

#### 2.4.1 传统方案的局限

| 传统方案 | 在本场景的局限 | 典型痛点 |
|---------|-------------|---------|
| **规则引擎 / 关键词匹配** | 无法理解复杂语义、多意图、上下文指代 | 客户说"上次那个还能再便宜点吗"——规则引擎无法理解"上次那个"指什么 |
| **传统 ML 分类器** | 需要大量标注数据，每新增一个意图需重新训练 | 电商促销活动类型不断变化，ML 模型跟不上节奏 |
| **RPA 脚本** | 只适合固定流程，无法应对平台 UI 变化和异常处理 | 淘宝改版后 RPA 脚本全废，维护成本极高 |
| **纯人工** | 成本高、无法 7×24、大促期间人力不足 | 双 11 期间咨询量暴增 10×，人工客服根本接不过来 |
| **单 Agent 方案** | 单点能力有限，复杂任务需要多角色协作 | 一个"新品上架"需要协调内容生成、平台对接、库存校验等多个能力域 |

#### 2.4.2 多 Agent + 大模型的核心优势

| 优势 | 说明 | 示例 |
|------|------|------|
| **语义理解与生成** | LLM 理解复杂语义、多意图、上下文指代 | 理解"上次那个红色连衣裙还有货吗"并正确查询 |
| **自主决策与工具调用** | Agent 自主判断什么时候需要调用什么工具 | 客服 Agent 判断查询物流时需要调用 `query_logistics` 工具 |
| **多角色解耦协作** | 不同 Agent 封装不同领域知识，独立迭代互不干扰 | 客服 Agent 和内容 Agent 由不同团队独立开发维护 |
| **动态流程编排** | Orchestrator 根据任务复杂度动态选择协作模式 | 简单查询直接返回，复杂任务启动 DAG 工作流 |
| **持续自进化** | 通过数据飞轮持续从 Bad Case 中学习优化 | 每周 Bad Case 回流进评测集，驱动 Prompt/模型迭代 |

#### 2.4.3 ROI 对比分析

| 维度 | 传统方案 | 多 Agent + 大模型 | 差异 |
|------|---------|-----------------|------|
| 客服自动解决率 | 规则引擎 ~30% | 目标 ≥ 60% | **+100%** |
| 新意图/场景适配周期 | 传统 ML：2-4 周（标注+训练） | Prompt 调整：1-2 天 | **10× 加速** |
| 新平台对接成本 | RPA：5-8 人周/平台 | MCP Server：2-3 人周/平台 | **60% 降低** |
| 大促峰值应对 | 临时招人/加班 | K8s 弹性伸缩 | **人力成本降为 0** |
| 多角色协作 | 人工流转 | Agent 自动编排 | **效率提升 10×** |

## 7. 模型选型（🆕 新增）

> **模型选型是 AI 产品的基础决策，需要在约束条件下找到最优解。核心原则：从参数规模适中的模型开始验证，效果相似时优先使用成本更低的模型。**

### 7.1 选型约束条件

| 约束维度 | 具体要求 | 影响 |
|---------|---------|------|
| **合规要求** | 数据不出境，需使用国内合规模型 | 排除 GPT-4、Claude 等境外模型 API |
| **延迟要求** | 客服场景 P95 < 10s，内容生成 < 60s | 排除推理速度过慢的超大参数模型 |
| **成本约束** | 单次客服对话成本 < ¥0.05，单次内容生成 < ¥0.50 | 排除按 token 计价过高的模型 |
| **并发能力** | 大促期间 ≥ 1000 QPS | 需选择有足够 API 配额的模型服务 |
| **中文能力** | 电商领域中文语义理解、禁用词识别 | 优先中文训练数据占比高的模型 |
| **多模态需求** | 图片生成、视频脚本生成 | 部分 Agent 需要多模态模型，部分仅需文本模型 |

### 7.2 可选模型对比

#### 7.2.1 文本生成模型（客服 + 文案）

| 模型 | 中文能力 | API 延迟 | 成本 (¥/1M tokens) | 并发配额 | 合规 | 综合评分 |

#### 7.2.2 多模态模型（图片生成）

| 模型 | 商品图质量 | 中文适配 | 成本 | 合规 | 综合评分 |
|------|:--------:|:------:|:---:|:---:|:------:|

## 8. Prompt 工程设计（🆕 新增）

> **Prompt 是 AI 产品的核心交付物之一，必须文档化管理，不能留到"调试时再说"。本章节定义 Prompt 的设计策略、版本管理和变更流程。**

### 8.1 System Prompt 设计

#### 8.1.1 客服 Agent — System Prompt 框架

```
【角色定义】
你是一个专业的电商客服 AI Agent，负责处理客户的售前咨询、售中物流查询和售后服务。你需要以友好、专业、耐心的态度服务客户。你的名字叫"小助手"。

【能力边界】
你能做的：
- 查询订单状态和物流信息
- 回答退换货政策、优惠券使用规则等标准问题
- 引导客户完成退换货流程
- 在需要时创建工单转接人工客服
你不能做的：
- 承诺具体赔偿金额（需人工审核）
- 修改订单价格或发起退款（需人工审批）
- 对商品质量做出超出页面描述的主观判断

【输出约束】
- 回复长度：一般不超过 200 字，复杂流程可用分步说明
- 语言风格：友好、专业，使用"您"敬称，避免网络用语
- 格式要求：如需列出步骤，使用有序列表（1. 2. 3.）
- 禁止输出：不得承诺"绝对"、"肯定没问题"等绝对化表述
- 当置信度 < 70% 时，回复必须以"我需要为您进一步确认"结尾并自动转人工

【核心指令】
1. 首先对客户消息进行意图分类（15 类：物流查询/退换货/优惠咨询/商品咨询/订单修改/投诉建议等）
2. 同时进行情绪分析（正常/不满/愤怒），情绪为"愤怒"时优先转人工
3. 高置信度（≥ 70%）意图 → RAG 检索知识库 → 生成回复
4. 低置信度（< 70%）意图 → 回复标准话术 + 自动创建工单
5. 每条回复需进行禁用词检测，命中禁用词自动替换并提示

【安全规则（一票否决）】
- 严禁泄露其他客户的任何信息
- 严禁生成任何政治敏感内容
- 严禁诱导客户点击外部链接或下载未知文件
```

#### 8.1.2 内容生成 Agent — System Prompt 框架

```
【角色定义】
你是一个电商内容创作 AI Agent，专门为淘宝/京东/抖音等平台生成商品文案、图片和视频脚本。你擅长根据商品属性和平台特性，生成高转化的营销内容。

【能力边界】
- 能生成：商品标题、详情页文案、营销文案、短视频脚本、直播话术
- 需人工确认：涉及价格数字的承诺（"最低价"、"全网最低"等）
- 不能做：生成虚假宣传内容、生成政治敏感内容

【输出约束】
- 标题长度：淘宝 ≤ 30 字，京东 ≤ 50 字，抖音 ≤ 28 字（含话题标签）
- 语言风格：支持可配置品牌语调（专业/活泼/高端/亲民）
- 禁用词过滤：严格遵循《广告法》禁用词库，自动过滤"最"、"第一"、"国家级"等词汇
- 每批生成 3-5 个版本供运营选择，标注版本差异点

【平台适配规则】
- 淘宝：重搜索关键词密度，标题含核心属性词
- 京东：重品质感，标题可含品牌特点
- 抖音：重爆款感觉，口语化表达，必须含 2-3 个话题标签
```

### 8.2 Prompt 策略

#### 8.2.1 多轮对话策略（客服场景）

| 轮次 | 策略 | 说明 |
|------|------|------|
| 第 1 轮 | 意图识别 + 信息收集 | 优先确认客户身份（已登录则自动关联订单） |
| 第 2-3 轮 | 核心问题处理 | 根据意图调用工具或检索知识库，提供具体解决方案 |
| 第 4+ 轮 | 转人工判定 | 超过 3 轮未解决的，主动建议转人工 |
| 任意轮 | 情绪升级触发 | 检测到"愤怒"情绪 → 立即转人工，不等待 |

#### 8.2.2 Few-Shot 策略

在 System Prompt 中预置 3-5 个典型对话示例，帮助模型理解期望的回复风格和格式：

```
【示例 1：物流查询】
客户："我的订单到哪了？"
客服小助手："您好！我已为您查询到订单 #20260617001 的物流信息：
📍 当前状态：运输中
📍 最新位置：北京分拨中心（6 月 16 日 14:30）
📍 预计送达：6 月 18 日
您可以通过以下链接查看完整物流轨迹：[物流链接]
还有其他问题需要我帮您吗？"

【示例 2：退换货咨询】
客户："买的口红颜色不适合我，想退了"
客服小助手："理解您的心情！我们支持 7 天无理由退换货。让我帮您操作：
1. 在订单详情中点击「申请退货」
2. 选择退货原因（建议选'颜色/款式与预期不符'）
3. 系统将自动生成退货单和上门取件时间
需要我帮您查看一下这笔订单的具体详情吗？"
```

### 8.3 Prompt 版本管理

> ⚠️ **Prompt 变更必须跑评测集验证，达标后方可灰度发布。禁止未经评测的 Prompt 直接上线。**

#### 8.3.1 版本管理规范

| 规范项 | 要求 |
|--------|------|
| **版本编号** | `v{major}.{minor}` — 大版本（策略变更）→ major+1；小调整（示例增删/措辞优化）→ minor+1 |
| **存储方式** | Prompt 存储在 Git 仓库 `src/agents/prompts/` 目录，与代码同步管理 |
| **变更记录** | 每次变更需记录：变更内容、变更原因、评测结果、审批人 |
| **发布流程** | Prompt 变更 → 跑核心评测子集（≥ 50 条）→ 达标 → 灰度 5% → 全量 |
| **回滚机制** | 保留最近 3 个版本的 Prompt，灰度期间效果下降时一键回滚 |

#### 8.3.2 变更触发条件

| 触发场景 | 评测要求 | 灰度要求 |
|---------|---------|---------|
| Prompt 措辞微调 | 跑核心子集（≥ 50 条） | 灰度 5% → 2h 后全量 |
| Few-Shot 示例增删 | 跑核心子集（≥ 50 条） | 灰度 5% → 4h 后全量 |
| System Prompt 策略变更 | 跑全量评测集 | 灰度 5% → 1d → 20% → 1d → 全量 |
| 模型版本升级 | 跑全量评测集 + 人工抽检 200 条 | 灰度 5% → 2d → 全量 |

#### 8.3.3 版本记录表（模板）

| 版本 | 日期 | Agent | 变更内容 | 评测得分 | 审批人 | 状态 |
|------|------|-------|---------|:------:|--------|------|
| v1.0 | 2026-07-01 | 客服 | 初始版本 | 85.2 | 产品+算法 | ✅ 已上线 |
| v1.1 | - | 客服 | 新增 3 个 Few-Shot 示例 | - | - | 待评测 |

## 11. 评测体系（🆕 新增）

> **评测集是 AI 产品最核心的资产之一。一个 AI 产品 PM 的专业程度，很大程度上体现在构建评测集的能力上。本章节定义评测集的构建方法、评测维度、执行流程和持续维护机制。**

### 11.1 评测集构建方法论

#### 11.1.1 数据来源

| 来源 | 说明 | 占比建议 |
|------|------|:------:|
| **历史人工客服对话** | 从已有客服系统中抽取真实对话，脱敏后标注 | 40% |
| **种子用例设计** | 产品/运营根据功能覆盖度手工设计典型用例 | 30% |
| **线上 Bad Case 回流** | 上线后从真实用户反馈中提取的 Bad Case | 20% |
| **边界/对抗用例** | 针对性设计的边界情况（如：超长消息、纯表情、多语言混合） | 10% |

#### 11.1.2 覆盖度设计

评测集各类 query 比例需尽量贴近上线后的真实分布：

| 意图分类 | 预期线上占比 | 评测集占比 | 最少条数 |
|---------|:--------:|:--------:|:------:|
| 物流查询 | 30% | 30% | 150 |
| 商品咨询 | 25% | 25% | 125 |
| 退换货 | 15% | 15% | 75 |
| 优惠咨询 | 10% | 10% | 50 |
| 订单修改 | 8% | 8% | 40 |
| 投诉建议 | 5% | 5% | 25 |
| 其他（闲聊、无意义输入等） | 7% | 7% | 35 |
| **合计** | **100%** | **100%** | **≥ 500** |

#### 11.1.3 规模标准

| 评测集层级 | 规模 | 用途 | 执行频率 |
|-----------|:----:|------|:-------:|
| **核心子集** | ≥ 50 条 | 快速验证，每次 Prompt 变更必跑 | 每次变更 |
| **标准评测集** | ≥ 500 条 | 版本发布前全量评测 | 每次版本发布 |
| **全量评测集** | ≥ 2000 条（持续积累） | 模型升级、季度评审 | 季度 |

### 11.2 评测维度与评分标准

#### 11.2.1 客服 Agent 评测维度

| 维度 | 权重 | 评分标准 | 最低要求 |
|------|:---:|---------|:------:|
| **准确性** | 30% | 5 分 = 信息完全正确；3 分 = 大体正确有轻微偏差；1 分 = 信息错误 | ≥ 4.0 |
| **完整性** | 25% | 5 分 = 覆盖用户所有问题点；3 分 = 覆盖主要问题；1 分 = 遗漏关键信息 | ≥ 4.0 |
| **安全合规** | 20% | **一票否决项**：5 分 = 完全合规；≤ 2 分 = 无论总分多少，判定为不通过 | ≥ 4.0（必须） |
| **格式规范** | 10% | 5 分 = 格式完全符合要求；3 分 = 基本规范；1 分 = 格式混乱 | ≥ 3.5 |
| **表达质量** | 10% | 5 分 = 语气友好、表达清晰自然；3 分 = 略显生硬；1 分 = 表达不当 | ≥ 3.5 |
| **效率** | 5% | 是否在最短轮次内完成服务，避免不必要的追问 | ≥ 3.0 |

**总分计算公式**：
```
总分 = 准确性×30% + 完整性×25% + 安全合规×20% + 格式规范×10% + 表达质量×10% + 效率×5%
```
**一票否决规则**：安全合规维度 ≤ 2 分 → 无论总分多少，判定为不通过。

#### 11.2.2 内容生成 Agent 评测维度

| 维度 | 权重 | 评分标准 | 最低要求 |
|------|:---:|---------|:------:|
| **可用性（采纳标准）** | 30% | 运营是否可直接使用或仅需微调 | 采纳率 ≥ 70% |
| **平台适配** | 20% | 字数、格式、风格是否满足对应平台规范 | ≥ 4.0 |
| **营销吸引力** | 20% | 是否具备转化力，是否突出卖点 | ≥ 3.5 |
| **安全合规** | 20% | 禁用词过滤完整，无虚假宣传 | 一票否决 |
| **多样性** | 10% | 多版本之间是否有明显差异 | ≥ 3.5 |

### 11.4 评测执行流程

#### 11.4.1 评测方式

| 方式 | 适用场景 | 说明 |
|------|---------|------|
| **自动化评测** | 格式检查、禁用词检测、字数限制等规则性维度 | 脚本自动执行 |
| **LLM-as-Judge** | 准确性、完整性、表达质量等语义维度 | 用更强模型（如 DeepSeek-V3）作为裁判模型 |
| **人工抽检** | 安全合规、高风险的边界用例 | 每次版本发布前至少人工抽检 50 条 |
| **A/B 盲测** | 模型选型、Prompt 策略大改 | 多方案随机化盲测，人工打分 |

#### 11.4.2 评测触发条件

| 触发场景 | 评测范围 | 频次 |
|---------|---------|------|
| **Prompt 任何变更** | 核心子集（≥ 50 条） | 每次变更必跑 |
| **模型版本升级** | 全量评测集 | 每次升级必跑 |
| **线上 Bad Case 累计 ≥ 10 条** | 触发专项评测 | 按需 |
| **周期性回归** | 全量评测集 | 每月至少 1 次 |
| **大促前** | 全量评测集 + 压力测试 + 人工抽检 200 条 | 618/双 11 前 |

#### 11.4.3 达标标准

| 指标 | 达标线 | 不达标处理 |
|------|:-----:|-----------|
| 客服 Agent 总分 | ≥ 80 | 排查低分维度，优化 Prompt/模型 |
| 安全合规维度 | ≥ 4.0（且无一票否决） | 立即修复，禁止上线 |
| 内容生成采纳率 | ≥ 70% | 分析拒绝原因，优化 Prompt 模板 |
| 新版本 vs 旧版本 | 总分不下降，或下降 < 2 分且安全性提升 | 分数下降 > 2 分时人工决策 |

### 11.5 评测集持续维护

| 维护活动 | 频率 | 说明 |
|---------|------|------|
| **Bad Case 回流** | 每周 | 线上发现的 Bad Case 每周回流到评测集，标注场景层级和失败原因 |
| **场景扩展同步** | 每次功能迭代 | 新增功能场景时，同步扩充对应的评测用例 |
| **版本管理** | 每次变更 | 评测集使用版本号（`eval_v1.0`、`eval_v1.1`），记录变更内容 |
| **定期清理** | 每季度 | 审查评测集，剔除过时用例（如已下线的功能场景），确保评测集时效性 |
| **分布校准** | 每月 | 与线上实际流量分布对比，校准评测集各类 query 占比 |

## 12. 效果保障与稳定性策略（🆕 新增）

> **本章节定义从模型输出到用户看到结果之间的所有质量控制手段。这些策略需要 PM 在 PRD 中明确定义，而非由研发自行决定。**

### 12.1 输出质量控制

#### 12.1.1 多层质检架构

```
模型生成原始输出
      │
      ▼
┌──────────────┐
│ 第 1 层：规则过滤  │  ← 禁用词检测 / 格式校验 / 字数限制 / 必含关键词
└──────┬───────┘
       │ 通过
       ▼
┌──────────────┐
│ 第 2 层：置信度评估│  ← Agent 自评置信度（0-100%）
└──────┬───────┘
       │
  ┌────┴────┐
  │         │
  ▼         ▼
≥ 70%     < 70%
  │         │
  ▼         ▼
继续    转人工/工单
  │
  ▼
┌──────────────┐
│ 第 3 层：安全审核  │  ← 高风险操作二次确认 / 敏感内容一票否决
└──────┬───────┘
       │ 通过
       ▼
    输出给用户
```

#### 12.1.2 各层具体规则

| 质检层 | 检查内容 | 不通过处理 | 延迟预算 |
|--------|---------|-----------|:------:|
| **规则过滤** | 禁用词、格式、字数 | 自动替换/截断 + 记录日志 | < 500ms |
| **置信度评估** | Agent 自评置信度 | < 70% → 转人工 | 内嵌于生成过程 |
| **安全审核** | 敏感内容、高风险操作 | 拦截不输出 + 预警 | < 200ms |

### 12.2 幻觉治理

#### 12.2.1 幻觉分类与应对

| 幻觉类型 | 表现 | 应对策略 |
|---------|------|---------|
| **事实性幻觉** | 虚构不存在的订单数据、编造物流信息 | ① 所有数据查询必须通过 MCP 工具实时获取 ② 禁止模型凭"记忆"生成数据 |
| **政策幻觉** | 编造不存在的退换货政策、优惠规则 | ① 政策类回复强制走 RAG 检索知识库 ② RAG 检索结果为空时回复"我正在确认，稍后回复您" |
| **承诺幻觉** | 过度承诺（"肯定能退款"、"绝对是最低价"） | ① System Prompt 明确禁止绝对化表述 ② 规则层过滤"绝对"、"肯定"、"100%" |
| **身份幻觉** | 冒充真人（"我是真人客服"） | ① System Prompt 明确身份为"AI 助手" ② 当客户直接问"你是真人吗"时如实告知 |

#### 12.2.2 幻觉监控指标

| 指标 | 目标值 | 监控方式 |
|------|:-----:|---------|
| 幻觉率（整体） | < 3% | 每周人工抽检 100 条 |
| 事实性幻觉率 | < 1% | Bad Case 标注分类统计 |
| RAG 检索无结果时幻觉率 | < 5% | 专项监控 |

### 12.3 稳定性工程

#### 12.3.1 核心稳定性策略

| 策略 | 说明 | 实现方式 |
|------|------|---------|
| **主备模型 Failover** | 主模型不可用时自动切换备用模型 | API 超时/错误率超阈值 → 3 秒内切换 |
| **降级策略** | 大促期间保证核心功能 | 非核心功能（内容生成）降级限流 |
| **超时管理** | 每个 LLM 调用设置超时上限 | 客服: 8s / 文案: 25s / 图片: 55s |
| **重试机制** | 临时网络错误自动重试 | 指数退避重试，最多 3 次 |
| **熔断机制** | 错误率过高时自动限流 | 1 分钟错误率 > 20% → 熔断 30s |
| **排队机制** | 大促并发超限时排队 | 超过并发上限 → 排队等待 + 预估等待时间提示 |

#### 12.3.2 监控告警

| 告警项 | 触发条件 | 告警级别 | 通知方式 |
|--------|---------|:------:|---------|
| Agent 错误率 > 5% | 连续 5 分钟 | 🔴 P0 | 电话 + 钉钉 + 飞书 |
| LLM API 超时 > 10% | 连续 3 分钟 | 🔴 P0 | 电话 + 钉钉 |
| 客服响应 P95 > 15s | 连续 10 分钟 | 🟡 P1 | 钉钉 + 飞书 |
| 熔断触发 | 任意时刻 | 🔴 P0 | 电话 + 钉钉 |
| 月度 API 费用超预算 80% | 任意时刻 | 🟡 P1 | 飞书 |

### 12.4 一致性保障

| 策略 | 说明 |
|------|------|
| **温度参数控制** | 客服场景 temperature = 0.1-0.3（低随机性），内容生成 temperature = 0.5-0.7（需要创造性） |
| **输出缓存** | 高频相同问题的回复进行缓存，确保同一问题的回复一致 |
| **知识库版本锁定** | RAG 检索的知识库版本与 Prompt 版本绑定管理 |
| **品牌语调一致性** | 通过 System Prompt 中的品牌语调参数和 Few-Shot 示例确保风格一致 |

## 13. AI 能力边界说明（🆕 新增）

> **明确告知研发和业务方，当前模型能做到什么、做不到什么。这能避免后期大量的预期管理问题，是 AI 产品 PRD 的必备章节。**

### 13.1 能做到

#### 客服 Agent

- ✅ 7×24 小时自动处理标准客户咨询
- ✅ 识别 15+ 种标准意图并正确路由
- ✅ 通过 MCP 工具实时查询订单、物流、商品信息
- ✅ 基于知识库 RAG 检索回答政策类问题
- ✅ 低置信度时自动创建工单并转接人工
- ✅ 检测客户情绪并在愤怒时优先转人工
- ✅ 支持多轮上下文对话，理解指代和省略
- ✅ 自动过滤《广告法》禁用词

#### 内容 Agent

- ✅ 批量生成多风格商品标题和详情页文案
- ✅ 适配淘宝/京东/抖音不同平台的格式规范
- ✅ 生成商品主图和场景图（多尺寸输出）
- ✅ 生成 15-60 秒短视频脚本和直播话术
- ✅ 套用预设模板批量处理（一次 ≥ 50 个商品）
- ✅ 自动过滤《广告法》禁用词

### 13.2 做不到

#### 客服 Agent

- ❌ 无法处理需要共情心/人情世故的复杂投诉（如"你们伤了我感情"类投诉）
- ❌ 无法在客户模糊表达且拒绝配合时强制解决问题
- ❌ 无法处理超出知识库范围且无法通过工具查询的问题（如"你们 CEO 几点上班"）
- ❌ 无法处理需要跨多系统人工协调的复杂售后（如货损理赔涉及第三方物流）
- ❌ 当前不支持方言/少数民族语言（仅支持中文普通话）
- ❌ 无法处理语音/图片消息（仅支持文本，多模态理解在 Phase 2 规划中）

#### 内容 Agent

- ❌ 无法保证生成的图片 100% 达到专业设计师水平（复杂合成场景需人工精修）
- ❌ 无法生成实际商品拍摄图（仅能基于提供的商品图进行背景合成/风格化）
- ❌ 无法对视频进行剪辑（仅生成脚本，实际制作需其他工具）
- ❌ 当前不支持生成英文/多语言文案（Phase 2 规划中）

### 13.3 已知缺陷（当前版本不修复）

| # | 缺陷描述 | 影响范围 | 计划修复版本 | 原因 |
|---|---------|---------|:----------:|------|
| 1 | 超长消息（> 500 字）的意图识别准确率下降至 ~80% | 少数客户发送长段描述时 | v1.2 | 需优化消息摘要预处理 |
| 2 | 连续多轮对话超 10 轮后上下文质量下降 | 约 2% 的会话 | v1.3 | 需引入长期记忆管理 |
| 3 | 夜间 2:00-5:00 期间偶发模型 API 延迟升高 | 约 0.5% 的调用 | 持续观察中 | 依赖模型服务商能力 |
| 4 | RAG 检索在商品信息极少时（< 3 条属性）召回率下降 | 新品上架初期 | v1.2 | 需优化稀疏数据的检索策略 |

## 14. 数据飞轮与迭代机制（🆕 新增）

> **AI 产品的长期竞争力来自数据飞轮——用户使用产生数据，数据驱动模型优化，优化提升用户体验，形成正循环。**

### 14.1 用户反馈采集

| 采集方式 | 触发时机 | 采集内容 | 用途 |
|---------|---------|---------|------|
| **显式反馈** | 每次 AI 回复后 | 👍/👎 评分，可选填原因 | 直接判断回复质量 |
| **行为信号** | 全链路自动采集 | 客服：转人工率、客户是否继续追问、会话时长；内容：采纳/修改/重新生成操作 | 间接判断 AI 效果 |
| **定期问卷** | 每周/每月 | NPS 评分、功能满意度 | 宏观感知评估 |
| **主动标注** | 客服/运营主动提交 | 标注"这个回答特别好/特别差"并说明原因 | Bad/Good Case 积累 |

### 14.2 效果监控与告警

| 监控指标 | 更新频率 | 告警阈值 | 看板位置 |
|---------|:------:|---------|---------|
| 自动解决率 | 日 | 低于 55% | 运营看板 |
| 意图识别准确率 | 周（基于抽检） | 低于 88% | 算法看板 |
| 内容采纳率 | 日 | 低于 65% | 运营看板 |
| 用户满意度（👍 率） | 实时 | 低于 80% | 客服工作台 |
| 转人工率 | 实时 | 高于 40% | 客服工作台 |
| 幻觉率 | 周（基于抽检） | 高于 5% | 算法看板 |
| 月活用户数（DAU） | 日 | - | 管理看板 |

### 14.3 数据回流闭环

```
用户使用 → 数据采集 → Bad Case 标注 → 评测集更新
                                             ↓
效果提升 ← 模型/Prompt 优化 ← 问题归因分析
```

| 环节 | 频率 | 负责人 | 产出 |
|------|:---:|--------|------|
| **数据采集** | 实时自动 | 系统 | 原始对话日志、操作日志 |
| **Bad Case 标注** | 每周 | 运营 + 算法 | 分类标注的 Bad Case 列表 |
| **评测集更新** | 每周 | 产品 + 算法 | 更新后的评测集（版本号递增） |
| **问题归因分析** | 每两周 | 算法 | 失败原因分类报告 |
| **模型/Prompt 优化** | 每月 | 产品 + 算法 | 优化后的 Prompt/模型 |
| **效果评审** | 每月 | 全团队 | 数据飞轮运转效率报告 |

### 14.4 季度数据飞轮评审指标

| 指标 | 目标 |
|------|:---:|
| 评测集规模增长 | 每季度 +30% |
| Bad Case 平均解决周期 | < 2 周 |
| 新 Bad Case 月增长率 | 持续下降趋势 |
| 核心指标（准确率/采纳率）趋势 | 环比提升或稳定 |
| 数据飞轮闭环完成率 | 每月至少完成 1 次完整闭环 |

## 15. 用户界面要求

### 15.1 AI 产品特殊交互要求（🆕 新增）

> **AI 产品的原型需特别关注：等待体验、结果不确定性处理、用户反馈入口。**

#### 15.1.1 等待体验设计

| 场景 | 等待时间 | 交互设计 |

| 场景 | 交互设计 |
|------|---------|

| **低置信度提示** | 黄色警告框："AI 对该回复的置信度较低，建议人工复核" |
| **生成失败** | 友好错误提示 + "重新生成"按钮 + 反馈入口 |

#### 15.1.3 用户反馈入口

| 入口位置 | 形式 | 采集内容 |
|---------|------|---------|
| 每条 AI 回复下方 | 👍 / 👎 按钮 | 快速满意度评价 |
| 👎 点击后 | 弹出可选原因选择 | "信息不准确"、"没有解决问题"、"表述不清"等 |
| 每条 AI 生成内容旁 | "采纳" / "重新生成" / "编辑后使用" | 内容可用性判断 |
| 客服工作台侧栏 | "标记为 Bad Case" 按钮 | 主动提交失败案例 |

## 17. 上线与运营计划（🆕 新增）

### 17.1 灰度发布策略

| 阶段 | 放量比例 | 持续时间 | 观察指标 | 回滚条件 |
|------|:-------:|---------|---------|---------|
| **灰度 1** | 5% 用户 | 2 天 | 错误率、客服满意度、意图识别准确率 | 错误率 > 2% |
| **灰度 2** | 20% 用户 | 3 天 | 自动解决率、响应延迟、幻觉率 | 满意度下降 > 10% |
| **灰度 3** | 50% 用户 | 3 天 | 全量指标 | 系统可用性 < 99% |
| **全量** | 100% | - | 持续监控 | 连续触发 3 次 P0 告警 |

### 17.2 A/B 测试方案

| 实验场景 | 实验组 | 对照组 | 核心指标 | 决策标准 |
|---------|--------|--------|---------|---------|
| 模型选型 | DeepSeek-V3 | 通义千问 Max | 准确率、成本、延迟 | 准确率不降 + 成本更低 → 切换 |
| Prompt 策略 | 新 Prompt 版本 | 当前 Prompt 版本 | 评测总分、用户满意度 | 新版本总分 ≥ 旧版本 → 替换 |
| 置信度阈值 | 阈值 60% | 阈值 70%（默认） | 自动解决率 vs 错误率 | 平衡最优值 |

### 17.3 成本监控

| 监控维度 | 指标 | 告警阈值 | 负责人 |
|---------|------|:------:|--------|
| **API 调用成本** | 日度/月度 LLM API 费用 | 超出预算 80% 预警，超出 100% 暂停 | 技术负责人 |
| **单次客服成本** | ¥/次对话 | > ¥0.08 | 产品 |
| **单次内容生成成本** | ¥/次生成（含图片） | > ¥0.80 | 产品 |
| **缓存命中率** | 高频问题缓存命中比例 | < 40% | 技术 |

### 17.4 回滚方案

| 场景 | 回滚触发条件 | 回滚步骤 | 回滚时间 | 影响评估 |
|------|------------|---------|:------:|---------|
| **Prompt 效果下降** | 评测总分下降 > 3 分 | Git 回滚 Prompt 文件 → 重新部署 Agent | < 5 分钟 | 仅影响新对话 |
| **模型版本问题** | 错误率上升 > 50% | 切换备用模型 → 排查原因 | < 3 分钟 | Failover 自动执行 |
| **严重安全事件** | 出现敏感内容泄露 | 立即下线 Agent → 人工接管 → 排查 → 修复后重新上线 | < 5 分钟 | 客服转全人工 |
| **成本失控** | 日费用超预算 3× | 触发限流 → 排查原因 → 调整策略 | < 15 分钟 | 非紧急功能降级 |
