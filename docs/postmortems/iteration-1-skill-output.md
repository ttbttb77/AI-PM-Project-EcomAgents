## 三、偏差归因

### 偏差 1：前端 Dashboard 未接入 WebSocket

**Why 1**：后端 WebSocket 已实现，但前端 Dashboard 仍是静态 HTML，没有 JS 连接逻辑。
**Why 2**：前端 10 个页面全部手写，Dashboard 在 Phase 2 早期完成，那时后端还没设计 WebSocket 接口。
**Why 3**：前后端开发顺序是"先前端 → 后后端"，而不是"先 API 契约 → 并行开发"。
**Why 4**：没有提前定义 WebSocket 消息格式的契约文档。

**根因**：开发顺序不当（先前端后后端），缺少 API 契约先行
**根因类型**：流程问题（可控）
**责任归因**：架构设计（API 契约未前置）

---

### 偏差 2：真实 LLM API 未接入

**Why 1**：Demo 阶段优先验证架构和流程，模拟回复足够跑通演示。
**Why 2**：DeepSeek API Key 未配置，国内模型接入需要额外的合规审查。
**Why 3**：模拟回复的开发成本远低于真实 API 接入（无需处理超时/重试/降级/计费）。

**根因**：Demo 阶段优先级决策——架构完整性 > AI 真实性
**根因类型**：资源问题（部分可控）
**责任归因**：产品（MVP 范围决策）

---

### 偏差 3：Mock 数据真实度不足

**Why 1**：种子数据按线性趋势生成，未加入周末/促销/节假日波动。
**Why 2**：Mock 数据生成脚本只花了 5 分钟写，没有做"数据真实感校验"。
**Why 3**：没有真实的电商销售数据作为基线参考。

**根因**：Mock 数据设计阶段缺少真实业务场景映射
**根因类型**：设计问题（可控）
**责任归因**：数据设计（未参照电商行业基准）

---

### 偏差 4：并行任务进度条偶现跳跃

**Why 1**：3 个并行 Agent 使用固定 `setTimeout` 模拟，没有随机抖动（jitter）。
**Why 2**：初始版本没有考虑到"3 个任务同时完成"时的渲染冲突。
**Why 3**：CLI 的进度条实现用 `process.stdout.write` 覆盖行，并发写入时未加锁。

**根因**：并发渲染的边界场景未测试
**根因类型**：质量问题（可控）
**责任归因**：开发（并发边界测试缺失）

## 四、经验沉淀

### ✅ 做对了什么（继续保持）

| # | 做对的事 | 具体表现 | 建议固化方式 |
|---|---------|---------|-----------|
| 1 | **PRD 先行** | v1.1 融合模板，19 章节覆盖了模型选型/Prompt/评测体系等 AI 专项 | PRD 作为所有后续工作的唯一真相源 |
| 2 | **原型驱动开发** | 10 页深色 SaaS 原型先于后端完成，统一视觉语言 | 每个 Phase 先出原型再写代码 |
| 3 | **前后端解耦** | Express 服务静态文件 + REST API + WebSocket，前端纯 HTML 可独立浏览 | 保持 API-first 架构 |
| 4 | **真实数据闭环** | SQLite + 31 天 Mock + 分析脚本，不是空壳 | 每个功能必须有数据埋点 |
| 5 | **文档完整性** | PRD ×2 + 调研 + RICE + 指标 + 埋点 + 复盘 | 文档是产品资产，不是负担 |

### ❌ 做错了什么（立即停止）

| # | 做错的事 | 造成的后果 | 根因 |
|---|---------|----------|------|
| 1 | **先写前端再设计 API** | Dashboard 和后端 WebSocket 脱节 | 缺少 API 契约先行 |
| 2 | **3 个工作流同时铺开** | 每个都不够深，新品上架最完整但促销和日常运营较薄 | 应该先做透一个 |
| 3 | **品牌名随手起** | EcoAgents → EcomAgents，10 个文件全局替换 | 名字应该在 Phase 0 确定 |
| 4 | **数据基线建立太晚** | Phase 6 才开始分析，Phase 1-5 没有数据意识 | 应该 Day 1 就定义核心指标 |

### 💡 方法论沉淀

| # | 经验原则 | 适用场景 | 具体做法 |
|---|---------|---------|---------|
| 1 | **"API 契约先行"原则** | 前后端协作项目 | 在写任何代码前，先定义 API 的 Request/Response JSON 和 WebSocket 消息格式 |
| 2 | **"深度优先"原则** | 多工作流项目 | 先做一个工作流的完整闭环（前端→API→编排器→数据库→数据看板），验证通过后再横向扩展 |
| 3 | **"Day 1 基线"原则** | 数据驱动项目 | 项目启动时立即定义 3-5 个核心指标，从第一天就开始收集（哪怕是 Mock 数据） |
| 4 | **"命名即架构"原则** | 任何项目 | 项目名/品牌名/模块名在 Phase 0 确定，写入 `CLAUDE.md`，全局引用 |
